## 프로젝트 아이디어
웹에서 시나리오 초안을 입력하면 AI가 리뷰/개선 제안을 만들고, Notion의 제안 항목 상태를 `approved/rejected`로 바꾸는 방식으로 반영 여부를 결정하는 경량 제작 파이프라인을 구축한다.

## 목적
- 초안 작성부터 반영까지 반복 작업을 자동화해 제작 속도를 높인다.
- 개인 작업 + 최대 2~3명 협업에 맞는 단순한 승인 흐름을 유지한다.
- 최종 승인권(owner)을 명확히 해 세계관/톤의 일관성을 보장한다.
- 모바일/데스크톱 모두에서 같은 웹 기반 흐름을 사용한다.

## 범위(경량 원칙)
- 입력 채널은 웹 단일 채널로 운영한다. (텔레그램 제외)
- 승인 전용 별도 웹 대시보드는 만들지 않는다.
- Notion DB의 상태 필드 변경을 승인 트리거로 사용한다.
- 부분승인/복잡 권한/다단계 결재는 MVP 범위에서 제외한다.

## 핵심 기능
### 1. 웹 초안 입력
- **입력 채널**: 웹 폼(모바일/데스크톱 공용)
- **입력 데이터**: 챕터 ID, 초안 본문, 세계관 메모, 요청 타입(리뷰/확장/톤 정리)
- **처리 방식**: 제출 즉시 작업 ID 발급 후 큐 적재

### 2. AI 병렬 리뷰(경량)
- **MVP 모델 수**: 2개 모델 병렬 호출
- **출력 형식**: 공통 JSON 스키마(강점/문제점/수정 제안)
- **안전장치**: 타임아웃, 1회 재시도, 실패 시 부분 성공 허용

### 3. 제안 생성(섹션 단위)
- 원문 + 리뷰 결과를 통합해 섹션 단위 제안안을 생성한다.
- 제안안은 Notion의 "제안 항목"에 기록한다.
- 출력은 "변경 이유 + 적용 텍스트(또는 추가 텍스트)" 중심으로 단순화한다.

### 4. 승인/반려(Notion 상태 기반)
- **승인 UI**: Notion DB 상태값 변경으로 대체 (`pending -> approved/rejected`)
- **승인 정책**: owner 1인 승인(기본)
- **반영 방식**:
  - `approved`: 본문 문서의 해당 섹션 반영
  - `rejected`: 반려 사유 저장 후 재생성 대기
- **추적성**: 제안 버전, 상태 변경자, 시각, 반려 사유 로그 저장

## 권장 아키텍처(간소화)
### A. 처리 흐름
- Web Form -> API -> Queue -> Worker(AI 호출/통합) -> Notion(제안 생성)
- 상태 전이: `queued -> reviewing -> awaiting_approval -> applied/rejected`

### B. 저장 전략
- **최종 문서**: Notion 본문 페이지
- **제안/상태 관리**: Notion DB
- **운영 로그**: 경량 DB(SQLite 또는 Postgres)
- **백업**: 주기적 Markdown export를 Git에 저장

### C. 중복 반영 방지
- 승인 이벤트는 제안 버전 기준으로 1회만 처리한다.
- 이미 처리된 버전은 재반영하지 않는다(idempotency).

## 협업 모델(2~3명 기준)
- `owner`: 승인/반려 권한
- `editor`: 초안 작성/수정, 제안 확인
- 필요 전까지 추가 역할/복잡 권한 매트릭스는 도입하지 않는다.

## 스페이스 오페라 가이드라인
- 세력 갈등, 항해 스케일, 개인 서사의 균형을 유지한다.
- 설정 설명 과다를 줄이고 장면 중심으로 전개한다.
- 용어집(함선/기관/항성계)과 연대표를 기준 문서로 관리한다.

## 단계별 실행 계획
### Phase 1. MVP (1주)
- 웹 입력 + 큐 + 2모델 병렬 리뷰 구현
- Notion 제안 DB 생성 및 상태(`pending/approved/rejected`) 운영
- `approved` 시 섹션 반영, `rejected` 시 사유 저장

### Phase 2. 안정화 (1주)
- 재시도/타임아웃/비용 상한 적용
- 버전 체크 및 중복 반영 방지 강화
- Git 백업 자동화(일 1회 또는 작업 N건 단위)

### Phase 3. 선택 확장 (필요 시)
- 모델 수 확장(3~4개)
- 간단한 웹 작업 목록/상태 조회 화면 추가
- 부분승인 기능 검토

## 검증 지표
- 초안 입력부터 제안 생성까지 평균 시간(목표: 5분 이내)
- 승인 후 본문 반영 성공률(목표: 99% 이상)
- 재시도 후 처리 성공률
- 승인까지 필요한 평균 라운드 수

## 리스크 및 대응
- 모델 의견 충돌: 보수적 병합 규칙 + 근거 요약 제공
- 비용 상승: 모델 수 고정(MVP 2개) + 요청당 토큰 상한
- Notion 반영 실패: 재시도 큐 + 실패 로그 + 수동 재처리 버튼(추후)
- 중복 승인: 버전 체크 + 1회 처리 키로 차단

## 초기 운영 규칙(확정안)
- 입력은 웹 단일 채널로 운영한다.
- 최종본은 Notion 본문 문서에 저장한다.
- 제안 반영은 owner 승인(`approved`) 후에만 수행한다.
- 승인 전 원문 자동 덮어쓰기는 금지한다.
